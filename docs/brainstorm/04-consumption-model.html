How Stacks Get Consumed

Two consumers: humans pulling stacks, and agents discovering them at runtime

Human Consumption (Phase 1-2)
## Scaffold a new project using a stack
$ as init --stack littleknifelabs/kubernetes@2.1
→ creates project with profiles/, environments/, skills/ from stack
→ writes stacks.lock pinning exact version + digest
→ generates CLAUDE.md pointing agent at the skills

## Pull a stack into an existing project
$ as pull littleknifelabs/openstack@1.3
→ downloads stack to .stacks/littleknifelabs/openstack/1.3.0/
→ updates stacks.lock

## Upgrade with deprecation warnings
$ as upgrade openstack
→ shows diff of what changed
→ warns: "deploy-with-kubeadm deprecated in 2.1, use deploy-with-k3s"
→ updates stacks.lock

## Check for issues
$ as doctor
→ validates stack.yaml, profiles, environment schemas
→ checks for deprecated skills in use
→ verifies dependency compatibility
Agent Consumption (Phase 3)

The MCP Server

An MCP server that any AI agent can connect to. Exposes the registry as tools the agent can call.

// Tools the agent gets:
search_stacks(query)
"I need to deploy OpenStack"
→ ranked results with descriptions

get_stack_info(name, version)
→ manifest, skill list, profiles, deps

get_skill(stack, skill_name)
→ full skill content (markdown)

pull_stack(name, version, path)
→ downloads stack locally

The Flow

1
Agent hits a task it doesn't know how to do
"Deploy an OpenStack cloud"
2
Calls search_stacks("openstack deployment")
Gets back: littleknifelabs/openstack@2.1
3
Calls get_stack_info to evaluate
Reads skills, profiles, requirements
4
Asks human: "I found an OpenStack stack. Install it?"
Human approves (or auto-approve if trusted)
5
Calls pull_stack, loads skills into context
"I know OpenStack."
