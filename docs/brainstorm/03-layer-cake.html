The Three Layers of Agentic Stacks

Each layer builds on the one below. Ship bottom-up.

Layer 3 — Ship Last

Agent Discovery Protocol

The "I know kung fu" API. Agents find and pull stacks at runtime.

Phase 3
What it is:
  • MCP server that agents can connect to
  • Semantic search: "I need to deploy a K8s cluster"
  • Returns stack metadata + install instructions
  • Agent evaluates, pulls, loads skills into context
Why it matters:
  • Agents as consumers, not just humans
  • Self-assembling competence
  • Nobody else has this
  • Only works if layers 1+2 are solid
Layer 2 — Ship Second

Registry & Website (agentic-stacks.com)

Discover, browse, pull versioned stacks. Human-facing.

Phase 2
What it is:
  • Web UI: browse stacks, read docs, see versions
  • OCI-based distribution (push/pull via GHCR)
  • CLI: as pull openstack@2.1
  • Versioning: SemVer, lock files, immutable releases
  • Trust: verified publishers, signed artifacts
Why it matters:
  • Discovery — people find what they need
  • Distribution — reliable, versioned delivery
  • Community — others can publish stacks
  • Gravity — more stacks = more users
Layer 1 — Ship First

Stack Specification + Thin Runtime

The standard for what a stack is and how it works.

Phase 1
The Spec:
  • stack.yaml manifest format
  • Directory conventions (skills/, profiles/, environments/)
  • Profile merge rules & category ordering
  • Environment declaration schema
  • Skill composition patterns
  • Action flow (intent → plan → approve → execute → verify)
The Thin Runtime:
  • Profile loader + ordered deep merge
  • Schema validation
  • Config diff engine
  • Append-only state store
  • Approval gate engine
  • Small API surface, rarely changes
