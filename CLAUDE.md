## What This Is

Agentic Stacks — a platform for packaging, distributing, and discovering domain expertise for AI agents. A "stack" is a git repo containing skills (markdown that teaches an agent how to operate in a domain), plus metadata. The agent reads the skills and guides users through complex tasks like deploying OpenStack, bootstrapping Kubernetes, etc.

## Architecture

Four packages under `src/`, built with hatchling:

**`agentic_stacks`** — Core runtime library. Manifest parsing (`manifest.py`), config diffing (`config_diff.py`), state store (`state_store.py`), approval gates (`approval.py`). Optional: profile loading + deep merge (`profiles.py`), environment validation (`environments.py`).

**`agentic_stacks_cli`** — Click-based CLI. Key commands: `init`, `pull`, `search`, `update`, `publish`, `create`, `doctor`, `list`, `remove`, `login`. Config at `~/.config/agentic-stacks/config.yaml`.

**`web`** — FastAPI website and API. Routes in `routes/`: stacks.py (JSON API), namespaces.py (JSON API), web.py (HTML pages, sitemap, llms.txt, robots.txt). No database — reads from `formula_data.py` bundled at deploy time.

**`agentic_stacks_mcp`** — MCP server exposing stack discovery and pull as agent-callable tools.

## Distribution Model

Stacks are git repos. The registry (`agentic-stacks/registry`) holds formula YAMLs that index them.

- `init [path]` creates a user project with `stacks.lock`, `.gitignore`, and a `CLAUDE.md` pointing to `.stacks/*/CLAUDE.md`
- `pull` clones a stack's GitHub repo into `.stacks/<name>/` (or `git pull` to update)
- `pull <ref>` adds a new stack to the project and updates `stacks.lock`
- `update` checks for and pulls latest versions
- `list` shows all stacks in the current project
- `remove <ref>` removes a stack from `stacks.lock` and deletes its `.stacks/` directory
- `publish` generates a formula YAML for submission to the registry
- `search` queries the local registry cache (git-backed)
- `create owner/name [path]` scaffolds a new stack (for stack authors)

Default GitHub org is `agentic-stacks`. `pull openstack-kolla` resolves via registry formula or falls back to `github.com/agentic-stacks/openstack-kolla`.

## User Projects

Projects can compose multiple stacks for cross-domain expertise:

```bash
agentic-stacks init my-deployment
cd my-deployment
agentic-stacks pull kubernetes-talos
agentic-stacks pull hardware-dell
agentic-stacks list                  # see all stacks
```

## Website (src/web/)

FastAPI app serving HTML pages and JSON API. No database — formula data is bundled as a Python dict at deploy time.

- `db_formulas.py` — `FormulaDB` reads from in-memory formula list
- `formula_data.py` — auto-generated by `scripts/generate_formula_data.py` at deploy
- Templates use Tailwind CSS (CDN) and HTMX

## Cloudflare Workers Deployment

Production runs on Cloudflare Python Workers via ASGI bridge.

- `wrangler.toml` — config
- `src/main.py` — Worker entry point. Lazy imports, FormulaDB.
- Deploy workflow clones the registry repo, generates `formula_data.py`, then deploys
- Registry sync workflow triggers a deploy after updating formulas

**Workers constraints**: no threads, all route handlers must be async, heavy imports must be lazy.

## Testing

pytest. Tests in `tests/`, fixtures in `tests/fixtures/`. Web tests use `FormulaDB` with sample data from `tests/helpers.py`. CI runs on Python 3.12 and 3.13.
